21年2月第2篇周志

发表于2021-02-13|更新于2021-02-13|周志
|字数总计:4k|阅读时长:12分钟

近代音乐史包括三个时期，巴洛克时期，古典时期，浪漫时期。

由停滞而坎坷起步，严谨到了极点而复归于绚烂。

康塔塔情感单一，人本精神缺位，比较难受。

十九世纪初的音乐，植根于资本主义人文精神的理想性之沃土，兼具了宏大与细腻，此时期的音乐个人觉得最为优美。

个人主义的兴起，人的重新发现，无疑会使得个人情绪的表达逐渐成为下一个时代的主流…

至于一些先锋音乐…实在欣赏不来。

人生苦短，没时间浪费于一些事上。

过年了，除夕晚上10点多，从自习室回家，隆隆的爆响声隐约可闻，街旁无生命事物的轮廓赤裸裸于月光之下，自行车行驶于正道与歧途的夹缝之间。

不想回家。

与父亲长久的冷战自然是原因之一，更重要的，恐怕是某种难言的紧迫感。

没时间了。

虚度时间是青年人最奢侈的行为，正如永久睡眠是一个人所能祈求的最好的享受一样。因为可以逃开现实的折磨，而沉浸于一个精神上的梦幻之地。我已经没有这种奢侈了。

屋大维、亚历山大、阿尔弗雷德，皆是在青年时刻建立了自己的国家，而我的国度又在何处呢？

唯有加倍努力，来用疲劳麻痹自己，才能不因为内疚而感到折磨。

逃离家，家庭是令人窒息的绞索，一点点地，收紧，扼杀一个人。

应该一辈子也不会考虑结婚的，因为对情感方面几无需求，却强烈要求独立与自由。

让我们来玩一个观念的游戏，如果我们探究我们每个行为的行动目的，最后总是会归化到唯一一个动机上去，这个动机必然是指向自己的。这是一个指向自己的地址的指针。

根本价值只是我们认知世界的工具，至于它的内容是什么，其实并不重要。

以下为本周的周志

阅读自动化流程

需求分析

  1. 所有资料都可以处理的方式。手机端、平板端所有的地方都能看到需要处理的资料，都能够批注与笔记。—>dropbox实现，通过云来同步，或者通过邮箱邮寄或者保留网址到云端来阅读网页。这样的话，手机端主要在做筛选工作，PC端来进行阅读与注释。但是手机端也可以用acrobat（安卓）来标记与书写，并把结果云同步。

  2. 检索需求必需。考虑用acrobat reader 在pdf上进行批注，长篇批注用typora进行整理与记录——>检索pdf上的笔记与注释，这样也不用导出，长篇大论则采用同名到同文件夹的方式来随堂更新

  3. 笔记另外存一份，便于归档。我倾向于存放pdf和配套的笔记到数据库中，另外单独拿出笔记进行归档。

  4. 电脑上看完的文件归档整理，没看完的文件存储到电脑的暂存folder；手机上看完没看完都放云端，分文件夹放。

最终方案

工具：acrobat reader + typora + （filecenter + dropbox + westernworld + inoreader）

acrobat 有多种平台的版本，可以很好地支持阅读pdf与在pdf上打笔记与批注，也能够方便地检索全目录的批注。

无论是手机端还是电脑端都能处理资讯。

typora配合picgo等插件，能够出色地进行长篇大论的编写与排版，并且将图片或者文件存储到github的repositories（picgo可不只是图床哦）做cite,另外只要学习一些简单的html或者CSS的语法，就能取得惊艳的效果。

同步各个平台的文件采用dropbox，由于被GFW ban了，所以需要westworld。

对于一些可以订阅的资讯与文件，也可以直接RSS；newletter一类的可以邮到inoreader的mail里去，这样也能及时订阅到重要信息。

上述未处理的信息会首先进入待处理文件夹。经过处理后，手机端放入云端，同步到电脑上归档；电脑端直接filecenter归档。

每年把数据备份到几个网盘中，之后有质量好的磁盘也可以考虑？（或者光盘？）

另外我希望能把markdown文档分到几个地方保管，有的文档比较”脏“，涉及到大量用不到的知识与名词，可能会干扰一些比较高质量文档的检索。

在网上找到的epub格式的书籍，可以简单地转化为pdf格式的文件，从而被处理。

科普：知识共享许可协议(CC) 4.0协议 摘录

前言：无论你是否注意到，你的身边有很多内容创作者正在使用一些你可能没有了解过的开源分享协议，本专栏是对于CC 4.0协议的科普。

CC协议（A Creative Commons License）的中文叫做知识共享许可协议，是允许他人分发作品的公共版权许可之一。2002年12月16日，美国非盈利性组织Creative Commons首次发布了CC协议。4.0版本发布于2013年11月25日。

CC官网的提供的有中文版本，但基本都是机翻，我结合英文原版加上自己的理解做了整理。事先说明：此类协议原版是英文提供的内容，条款太多且晦涩难懂，翻译成中文有可能表达的不够符合原意。

协议介绍

在介绍具体协议之前，我们先介绍一下两个概念：

  • 共享 — 在任何媒介以任何形式复制、发行本作品
  • 演绎 — 修改、转化或以本作品为基础进行创作

只要你遵守以下介绍的具体协议，许可人就无权收回你的这些权利。

CC4.0许可协议有不同具体版本，分别由4种条例组合而成。

不同组合中使用的条例和图标介绍如下：

此部分内容取自Wikipedia中关于“知识共享许可协议”的说明，有所修改，具体链接为https://zh.wikipedia.org/wiki/知识共享许可协议。其采用CC-BY-SA授权。

BY：署名

署名（英语：Attribution，by）您（用户）可以复制、发行、展览、表演、放映、广播或通过信息网络传播本作品。您必须按照作者或者许可人指定的方式对作品进行署名。

您（用户）必须给出适当的署名，并提供指向本许可协议（指你使用的CC协议）的链接，同时标明是否（对原始作品）作了修改。您可以用任何合理的方式来署名，但是不得以任何方式暗示许可人为您或您的使用背书。

通常情况下的CC协议必有这条，所以也就是必须进行署名，尊重原作者劳动成果。并且还要提供CC协议的链接，注明是否有所修改。其中最奇怪的一点在于“不得以任何方式暗示许可人为您或您的使用背书”，这里的“背书”来源于银行的票据业务，是指使用人不能明示或暗示观众：“我使用了许可人的作品，许可人支持我的创作，并且可以用许可人的声誉确保我的使用行为可信可靠”。

背书是指票据转让时原持有人在票据背面加盖自己的印鉴证明该票据真实有效、如果有问题就找我。它不是担保（票据就是他的，他不能为自己担保），而是证明或声明的意思。

背书用于社会事务，一般是指【声明支持某人或某事、为某人或某事发声】，**一个单位或个人对另一个单位或个人的行为表示支持，以自身的信誉度来证明另一个单位或个人也同样可信可靠。**而不是为另一单位或个人的行为进行担保的意思。万一那个人的行为真的出了问题，你不能从经济或法律上追究这个人的责任。这是跟银行业务有所区别的地方。

具体的，署名需要包括以下内容：

  • 包含任何版权声明（如适用）。如果作品自身带有作者的版权声明，版权声明必须保持不变，或在一种合理的方式下重新分发给媒介。
  • 引用作者的名字，网名或用户ID等。如果作品发布在互联网上，如果存在作者的个人档案页面，应附上页面的链接。
  • 引用作品的标题或名称（如适用），前提是存在标题或名称。如果作品发布在互联网上，应在重新分发时应附上原作品的标题或者名称。
  • 引用作品的CC许可协议。如果作品发布在互联网上，应引用作者的CC许可协议，且应附有到CC网站的链接。（网址和跳转链接均可）
  • 如果作品是一个派生作品或改编作品，除了以上几点外，还应该说明这是一个派生作品，如“这是[作者]的[作品]的芬兰语翻译版本”或“剧本在[作者]的[原作品]基础上改编”。

SA：相同方式共享

相同方式共享（英语：ShareAlike，sa）您可以自由复制、散布、展示及演出本作品；若您改变、转变或更改本作品，仅在遵守与本作品相同的许可条款下，您才能散布由本作品产生的派生作品。

带有SA的协议要求传播时也要遵循同样的CC协议进行，比如原作者注明了NC非商业性，则你在使用其作品进行内容创作时也当注明请其他人遵循CC协议中的NC规定，并且要带有SA标记以告诉其他人也应当遵循此CC协议。

NC：非商业性使用

非商业性使用（英语：Noncommercial，nc）您可以自由复制、散布、展示及演出本作品；您不得为商业目的而使用本作品。

这个很好理解，带有NC的均不能用于商业目的。但在原创者允许的情况下，仍然可以用于商业目的。例如：小明在他的网站上以包括“非商业性运用”要素的答应协议发布了他的拍摄著作。在这种情况下，小红能够打印格斯的这幅相片欣赏。但是，如果没有小明的答应，小红不得贩卖这张打印的相片。

关于参与“bilibili创作激励计划”的内容是否属于商业行为，以及在第三方平台录制和直播游戏内容是否侵权、以及其他涉及到的侵权问题，可以查看我的下一篇专栏《非法律专业人士对于一些侵权问题的理解》。

ND：禁止演绎

禁止演绎（英语：No Derivative Works，nd)您可以自由复制、散布、展示及演出本作品；您不得改变、转变或更改本作品。

也就是只能原原本本地传播本作品，不得对其内容进行任何修改。例如：小明在自己的一首歌曲上使用包括“制止演绎”要素的答应协议。小刚想要截取小明的歌曲片段，混合在他的著作中而产生出全新的歌曲。在这种情况下，没有小明的答应，小刚就不能这么做（除非他的歌曲构成合理运用）。

混合搭配这些条例可产生11种有效组合（ND和SA互斥，还有1种没有包括任何条款），因为98%的使用情况均要求署名，其中的5种缺乏BY的条款不再使用，故一共有6种常用的版权规定组合。

  1. CC BY：署名标示（BY）
  2. CC BY-SA：署名标示（BY）- 相同方式共享（SA）
  3. CC BY-NC：署名标示（BY）- 非商业性使用（NC）
  4. CC BY-NC-SA：署名标示（BY）- 非商业性使用（NC）- 相同方式共享（SA）
  5. CC BY-ND：署名标示（BY）- 禁止演绎（ND）
  6. CC BY-NC-ND：署名标示（BY）- 非商业性使用（NC）- 禁止演绎（ND）

这些组合的权利与要求都写在上面的具体组成条例之中，将不同条例的权利与要求组合起来即为对应的协议权利与要求。如果组合中没有该条例，则表明组合的权利与要求内容与该条例相反。例如没有表明NC即为可商用，没有表明ND即为可二次创作。

对于之前提到的没有包括任何条款的组合，称为不受限制地在全球范围内发布内容，即CC0。CC0相当于是放弃以上四种条例的CC授权，成为Public Domain（公有领域）的素材。

中科大实景课堂资料下载随笔

咱在学卡尔曼滤波的东西，需要看一些网课，比较来比较去，感觉张真人的课是最吼的。

于是跑到了中科大的实景课堂，来听张老师的课。

中科大的课有时间限制，非本校学生只能看十分钟。

……

太小气了。

闲来没事，检查网页元素，发现在网页source标签里内嵌了m3u8的资源地址。哦豁？

然后顺藤摸瓜，真就找到了MP4资源的完整下载地址…….

想不明白科大这是什么迷惑操作。

github1s – 为 GitHub +1s，使用在线 VS Code 打开 GitHub 上的代码 摘录

github1s 是一个非常有趣的项目，它可以让你在 1 秒内（俗称 +1s）通过在线版本的 VS Code 来打开 GitHub 上的代码，只需要在对应项目的 URL 后面加上 1s 即可。

这是一个很有创意、很赞的项目，只需要在对应的 GitHub 项目 URL 后面加上 1s，就能用在线版 VS Code 打开 GitHub 上的代码。

  • 安装在云端的 VS Code，随时随地在浏览器使用 IDE

github1s 如何使用？

非常非常非常简单，以 github1s 项目自己的代码为例，第一行为 github1s 在 GitHub 的项目地址，也就是 GitHub 提供的正经网页版，而在 github 后面直接加 1s，然后再访问，就能打开一个在线版本的 VS Code，并在这个 VS Code 中加载了全部的项目代码：

  • https://github.com/conwnet/github1s
  • https://github1s.com/conwnet/github1s

然后就在熟悉的 VS Code 界面中打开了代码，但注意只读。

+1s 是什么意思

github1s 项目的副标题是：One second to read GitHub code with VS Code.

所以这个 +1s 的意思就是 1 秒钟的意思。

最后，github1s 在 GitHub 开源，你也可以自己部署，比如服务器慢点的可以 +2s、+3s 等等。

最后的最后，不得不感叹一下，VS Code 真的很厉害。
